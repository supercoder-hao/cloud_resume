Building a Serverless Resumé Website Using AWS: A Cloud Engineer's Journey

tags: HTML, CSS, Javascript, Python, AWS S3, AWS Certificate Manager, AWS CloudFront, AWS Route 53, AWS DynamoDB, AWS API Gateway, AWS Lambda, AWS Identity and Access Management (IAM)

Creating a serverless resume website using AWS is an excellent way to showcase both technical skills and an understanding of modern cloud architecture. In this blog, I’ll walk you through how I built my own resume website using various AWS services—combining scalability, automation, and cost-efficiency, all while adhering to best practices.

The goal was simple: host my resume as a static website with dynamic features like a visitor counter, all powered by AWS services. By the end of the project, I had leveraged AWS S3 for static site hosting, CloudFront for distribution, Route 53 for custom domain DNS, DynamoDB as a NoSQL database, and Lambda with API Gateway to handle serverless backend operations. Here's a detailed breakdown of each step.

1. HTML & CSS: Structuring and Styling the Resume

The foundation of the website was my resume, written entirely in HTML. I focused on clean, semantic markup to ensure the content was well-organized and easy to parse.

With the HTML structure in place, I styled the resume using CSS. While I’m not a designer, I aimed for a clean and professional look, using simple typography, margins, and padding to enhance readability. My goal was to strike a balance between simplicity and elegance. I was able to source an open-source template from HTML5 UP, which I customized to suit my needs.

2. Hosting the Resume on AWS S3: Static Website Hosting

To host the resume, I turned to Amazon S3. S3 is a great choice for static site hosting, offering high durability and scalability at a minimal cost while remaining serverless.

Although the website was now live and could be accessed from an S3 bucket URL, security and performance still needed to be addressed.

3. Securing with HTTPS via AWS CloudFront and AWS Certificate Manager

With the resume live, I needed to ensure it was served securely over HTTPS. I began by obtaining an SSL/TLS certificate from AWS Certificate Manager. This certificate would be used to configure Amazon CloudFront as a content delivery network (CDN) in front of the S3 bucket. CloudFront not only provided HTTPS but also improved the website’s performance globally through caching.

4. Custom Domain with Amazon Route 53

To give my resume a professional touch, I registered a custom domain and linked it to the CloudFront distribution. Additionally, I employeed subdomain routing to redirect traffic from a simple www. subdomain to resume. subdomain.

5. Adding a Dynamic Visitor Counter

Next, I wanted to track how many visitors accessed my resume. For this, I implemented a simple visitor counter using JavaScript. The idea was to display the count dynamically on the webpage. However, I needed a backend to store and manage the visitor data.

For the visitor counter backend, I used Amazon DynamoDB, a fully managed NoSQL database. DynamoDB was ideal due to its low-latency performance and pay-per-use pricing model. This database was created with a table with a partition key for storing the visitor count.

Rather than communicating directly with DynamoDB from the frontend (which would expose my database), I created a serverless API using AWS API Gateway and AWS Lambda. Here’s the breakdown:

  • API Gateway: Acted as the front door for the backend, receiving HTTP requests from the JavaScript on the website.
  • AWS Lambda: Handled the logic of reading, updating, and writing visitor data to DynamoDB.
For the Lambda function, I chose Python with the boto3 SDK to interact with DynamoDB. The function was simple: it fetched the current count, incremented it, and stored the updated value. This setup allowed me to keep my architecture serverless, reducing the need for manual server management and ensuring scalability.

6. IAM Permissions for Lambda

To ensure the Lambda function could interact with DynamoDB, I needed to configure the necessary permissions. This was done using AWS Identity and Access Management (IAM). I created a custom IAM role with the required policies to allow the Lambda function to read and write to the DynamoDB table. This role was then attached to the Lambda function, granting it the necessary permissions.

7. Source Control and CI/CD Pipelines

No modern project is complete without version control and continuous deployment. Throughout the project, I maintained version control using Git and GitHub. This allowed me to track changes, collaborate with others, and roll back to previous versions if needed.

For continuous integration and deployment (CI/CD), I used GitHub Actions. This automated the deployment process, ensuring that changes pushed to the main branch were automatically deployed to the S3 bucket. The CI/CD pipeline also ran tests to validate the changes before deployment, reducing the risk of errors in the live environment.
